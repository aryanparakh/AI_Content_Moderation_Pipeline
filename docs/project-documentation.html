1. What is this project?

This is a multi-stage content-moderation pipeline that decides, automatically and at scale, whether a piece of user-generated content is safe to publish, should be blocked, or needs a human to look at it.

Content moderation at scale is an unsolved operational problem. Simple keyword filters and naive classifiers produce far too many false positives, can't tell the difference between a genuine threat and friendly banter, and give no auditable reason for their decisions. This system addresses all three: it uses a large language model (Claude) for nuanced, context-aware classification, a tunable policy engine so every platform can set its own bar, and an explainable decision record that users and legal teams can audit.

2. What is it used for? (Use cases)

  • Social platforms & forums — automatically screen posts, comments, and messages for harmful content before they reach other users.
  • Reducing moderator workload — clear violations are actioned automatically; only genuinely ambiguous cases reach a human, so the team scales sub-linearly with content volume.
  • Multi-tenant / white-label products — one engine serves many platforms, each with its own policy (a children's app and an adult community need very different rules).
  • Auditable / compliant moderation — every decision stores the exact offending text, the harm category, and the reasoning, so it can be defended to users, regulators, or legal teams.
  • A feedback loop for model improvement — human reviewers' final calls are captured as labelled data to improve the classifier over time.

3. The seven harm categories

Every piece of content is scored, with a confidence value, against each of these categories:

CategoryWhat it catches
Hate speechAttacks or dehumanisation based on a protected attribute (race, religion, gender, etc.).
HarassmentTargeted bullying, threats, intimidation, or doxxing aimed at an individual.
SpamScams, mass advertising, link-farming, unsolicited promotion.
MisinformationDemonstrably false or dangerous claims (health, elections, safety).
Graphic violenceVivid depictions or glorification of gore or physical harm.
Adult contentSexually explicit material or solicitation.
Self-harmContent encouraging, instructing, or expressing intent toward suicide or self-injury.

4. How it works

Content flows through four stages. Crucially, only the first stage calls the AI model — everything after it is pure, deterministic code, which makes the system fast, cheap to re-run, predictable, and easy to test.

┌─────────────────────────────────────────────────────────────┐ content ─▶ │ 1. CLASSIFIER (Claude / claude-opus-4-8) │ + context │ per-category confidence + exact offending span │ │ judged IN CONTEXT (platform, author history, thread) │ └──────────────────────────┬──────────────────────────────────┘ ▼ ┌─────────────────────────────────────────────────────────────┐ │ 2. POLICY ENGINE (per-platform, deterministic) │ │ category on/off · review threshold · block threshold │ │ · custom keyword rules │ └──────────────────────────┬──────────────────────────────────┘ ▼ ┌─────────────────────────────────────────────────────────────┐ │ 3. CONFIDENCE ROUTER │ │ confidence ≥ block → AUTO-BLOCK │ │ review ≤ conf < block → HUMAN REVIEW QUEUE │ │ confidence < review → AUTO-ALLOW │ └──────────────────────────┬──────────────────────────────────┘ ▼ 4. EXPLAINABLE DECISION (segment + category + reasoning + threshold fired) ▼ human reviewer can override → saved as labelled feedback

Stage 1 — Context-aware classification

The content, plus its context (which platform, the author's history, the conversation thread), is sent to Claude. The model returns a structured result: for each detected harm category a confidence score (0–1), a severity, the verbatim offending segment, and a one-line reason. Context matters enormously — "I'm going to kill you" between friends after a game is banter, but the same words sent to a stranger on a children's platform is a credible threat. The model is told to weigh exactly that.

Stage 2 — Policy engine (the tunable part)

Each platform has its own policy: every category can be turned on/off and has two thresholds — a review threshold and an auto-block threshold — plus optional custom keyword rules. The same classification result produces different outcomes under different policies.

Stage 3 — Confidence-based routing

Model confidence vs. policyRouting
≥ auto-block thresholdAUTO-BLOCK — actioned automatically
between review and blockHUMAN REVIEW — sent to the queue
< review thresholdAUTO-ALLOW — published

Stage 4 — Explainable decision & human review

Every decision is stored with the offending segment highlighted, the category, the model's reasoning, and exactly which threshold or rule fired. Items routed to review appear in a moderator queue with the full context and the AI's recommendation; the reviewer's final call is recorded — and any disagreement with the AI is captured as labelled feedback for improving the model.

5. Example: same words, different context & policy

Context decides

Statement: “I'm going to kill you, you're dead next round.”

• Gaming voice chat between friends → ALLOW

• DM to a 12-year-old after an argument, sender has prior strikes → BLOCK / REVIEW

Policy decides

Statement: an explicit adult short story.

• Adult / Mature community (adult content disabled) → ALLOW

• Children's platform (strict thresholds) → BLOCK

7. What was built (components)

  • Classifier — Claude call with a context-aware prompt and structured output.
  • Policy engine — applies per-platform thresholds + custom rules, produces explainable per-flag decisions.
  • Pipeline + router — orchestrates classify → policy → route → store.
  • REST API — endpoints for moderation, the review queue, reviews/feedback, and policy management.
  • React dashboard — three screens: Moderate (submit + see explained decision), Review queue (moderator workspace with overrides), Policies (sliders, toggles, custom rules).
  • Evaluation harness — a labelled test set + runner that checks every success metric (classification coverage, context-sensitivity, routing, explainability, policy sensitivity).

Same statement, two contextsIdentical text in banter vs. threat context yields different actions.

9. How to run it

Backend: cd backend → npm install → copy .env.example to .env and add your ANTHROPIC_API_KEY → npm run dev (port 4000).

Frontend: cd frontend → npm install → npm run dev → open http://localhost:5173.

Prove the metrics: cd backend → npm run evaluate (runs the labelled set through the real pipeline and reports pass/fail).
